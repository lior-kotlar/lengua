### Task A1 (= issue #150) — Prompt-store hardening (#80 / PR #143 follow-ups) · the one HIGH finding

The DB-backed prompt store's fail-safe covers **read** failures but not **render** failures.

- [ ] **A1.a (HIGH)** Guard template rendering: `lengua_core/prompts.py` feeds DB-overridden
      fragment text straight into `str.format(...)` (lines ~278, ~300–311). A malformed override
      (unknown `{placeholder}`, positional `{}`, stray `{`) raises inside every generation request
      → **500s across the app until the row is fixed**. Wrap each fragment render in
      try/except → fall back to `CODE_DEFAULTS` for that fragment + log loudly.
- [ ] **A1.b** Read-time validation in `app/prompt_store.py`: cross-check snapshot keys against
      `prompts.PROMPT_KEYS` (unknown keys are silently ignored today) and skip + warn on
      empty-string content (today `''` is a "valid" override that silently blanks a fragment,
      e.g. an empty `output_format` deletes the output-shape instructions).
- [ ] **A1.c** Add one integration test proving the wiring end-to-end: boot `create_app` with a
      store installed and assert an HTTP generation's system instruction contains DB-overridden
      text (the autouse offline-store fixture in `tests/conftest.py` currently excludes the prod
      wiring from every test, so a wiring regression would be invisible + silent).
- [ ] **A1.d** Fix the torn-assembly race: one prompt build calls `PromptStore.get` 2–4×, and a
      concurrent `warm()` can swap the snapshot mid-build, mixing two prompt versions in a single
      request. Read the snapshot once per build and pass it through.
- [ ] **A1.e** Decide the version-pinning path: `resolve(version>0)` / `read_pinned_prompt_from_db`
      have **zero production callers** — the advertised reproducibility/A-B path is unreachable.
      Either wire it (request param / feature flag) or trim it and fix the docs.
- [ ] **A1.f (optional, migration-gated)** `prompt_versions` has no `CHECK (content <> '')` or
      key-enum constraint. A1.b covers this at read time; a DB CHECK needs a migration → pause
      for owner review per protocol.

### Task A2 (= issue #151) — Language-picker follow-ups (#95 / PR #145)

- [x] **A2.a** Case-sensitivity mismatch: the web matches names case-insensitively, the API
      dedupes exact-match — a curated pick of "French" with an existing "french" creates a
      case-variant **duplicate language row**. **Fixed server-side**: `get_by_name` now matches on
      `lower(name)` (portable across SQLite/Postgres), so both callers — idempotent add and the
      rename-conflict guard — dedupe case-insensitively.
- [x] **A2.b** In-flight reset race: `AddLanguageForm.submit()`'s `onSuccess` unconditionally
      resets to the picker, clobbering whatever the user selected/typed after pressing submit on
      a slow connection. **Fixed**: the "Change" / "Back to list" affordances are disabled while
      `isPending`, so the user can't navigate away mid-flight.
- [x] **A2.c** E2E gap: every e2e spec drives the **custom fallback** path; the curated pick →
      submit flow (the feature's primary new flow) has zero end-to-end coverage. **Added** a
      curated pick→submit→remove e2e in `languages.spec.ts`.
- [x] **A2.d** Analytics semantics: the `curated` event flag is documented as picker-path
      provenance but implemented as a name-table lookup (custom-path adds of a curated-named
      language are misclassified). **Fixed** by threading the real submit path from the form
      (a `curated` field on `AddLanguageInput`) rather than a name lookup.

### Task A3 (= issue #152) — Home-tile percent edge (#146 / PR #148)

- [x] `progressPercent` uses `Math.round`, so intra-band progress ≥ 0.995 renders **"100% to B2"**
      while the band chip still says B1. Floor (or cap at 99) in `apps/web/src/lib/cefr.ts` and
      add the ≥0.995 boundary test (existing tests cover 0.62 / 0.555 / C2-top / unknown only).
      **Fixed 2026-07-11** (#152, self-merged): `progressPercent` caps a below-1 fraction that rounds
      up to 100 at 99 (cap-at-99, not floor — leaves 0.555 → 56 and every other percent unchanged; an
      exact 1.0 still reads 100). Added the ≥0.995 boundary tests (0.994 / 0.995 / 0.999 / 1.0).
